Skydimo LED MCP Server

Physical LED strip status indicator — controllable via Model Context Protocol

Overview

This MCP server connects AI coding tools to a physical Skydimo SK0124 RGB LED strip (54 LEDs on COM4, CH340 USB-Serial). It exposes tools and resources that allow an AI assistant to change the LED color/animation based on what it is doing — giving you a physical real-time indicator of AI activity.

Hardware Setup

  • LED Strip: Skydimo SK0124, 54 RGB LEDs
  • Connection: CH340 USB-Serial adapter on COM4
  • Baud rate: 115200
  • Protocol: Adalight (Ada + 0x00 + 0x00 + N_LEDS + [R,G,B] * N_LEDS)

Architecture

A background Python daemon (skydimo_daemon.py) opens COM4 directly and renders LED animations. It monitors .ai_state for mode changes and switches instantly. The MCP server simply writes to .ai_state — no daemon restart needed.

Available MCP Tools

ToolDescriptionParameters
set_led_state tool Change the LED strip to a visual mode mode (required): one of the 12 modes below
start_daemon tool Start the background COM4 daemon mode (optional, default idle)
stop_daemon tool Stop the daemon gracefully (sends off frame) none
get_status tool Query daemon status and current LED state none

Usage Examples

Set LED state (via tool call)

# AI assistant calls:
set_led_state({ "mode": "reasoning" })
# → LED shows rainbow flowing animation

Query status

get_status({})
# → Daemon: running
# → LED State: reasoning

Direct CLI (for debugging)

# Write state directly (bypasses MCP)
echo reasoning > C:\Users\alexr\skydimo\.ai_state

# Or use the provided CLI
python C:\Users\alexr\skydimo\skydimo.py reasoning

Graceful Shutdown

The daemon listens for a .skydimo_daemon.stop signal file. When stop_daemon is called, a stop signal is written, the daemon exits its render loop cleanly, sends one final off frame to the LEDs, and releases COM4.

Configuration

To use this MCP server with opencode, add the following to your opencode.jsonc: 

{
  "mcp": {
    "skydimo-led": {
      "type": "local",
      "command": ["python", "C:\\Users\\alexr\\skydimo-mcp\\server.py"],
      "enabled": true
    }
  }
}

Latency Analysis

PathMeasured Latency
set_led_state (MCP → bridge → subprocess → skydimo.py) ~62ms
Direct file write (.ai_state) ~0.24ms
Daemon file_listener polling interval 100ms (max delay before LED updates)
Total end-to-end latency ~162ms (imperceptible to human eye)

Conclusion: The delay is not an I/O bottleneck. Each MCP call spawns a Python subprocess (~62ms), then the daemon's polling thread picks up the change within 100ms. Total latency is ~162ms — effectively real-time. The subprocess overhead could be eliminated by writing .ai_state directly in the bridge, but at 62ms it's already imperceptible.
